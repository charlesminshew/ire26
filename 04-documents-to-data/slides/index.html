Four strategies for today

1 · Read & flag

Use Claude or ChatGPT to read a document and surface what a reporter should dig into further.

2 · Extract structured data

Strategy 1: Read & flag

You have a document. You don't know if it's worth your time. Let AI be the first reader.

The triage prompt Read this document as a journalist. List the three most significant facts a reporter should follow up on. For each one, note which page it's on and why it matters. Flag anything that looks inconsistent, missing, or unusual.
The comparison prompt Compare these two contracts/reports. What changed between version A and version B? Focus on anything that shifts money, liability, deadlines, or who benefits. Ignore formatting differences.

Strategy 1: the rules

  • ✅ Use AI as a first reader, not a final one. It flags things for you to verify — it does not verify them.
  • ✅ Ask for page numbers. If AI can't cite a page, it may be hallucinating the fact.
  • ✅ Treat it like a source. You wouldn't publish a tip without checking it. Same rule applies here.
  • ⚠️ It will miss things. Long documents exceed context limits. Break large docs into sections.
  • ⚠️ Never send sensitive documents to a cloud AI without knowing your organization's policy.

Strategy 2: Extract structured data

When you need data out of a PDF as a table — and you don't want to paste it into a cloud tool.

◆ PDFplumber

Python library. Extracts text, tables, and character positions from born-digital PDFs with high accuracy. Tables come out as lists of lists — easy to push into a spreadsheet.

◆ natural-pdf

Newer library built on top of PDFplumber. Adds AI-assisted layout detection — better at PDFs with complex column layouts or mixed content. Also works on scanned PDFs via OCR.

🔒 Both run entirely on your machine. Nothing is sent to a server. This matters for documents with PII or sensitive sources.

Strategy 2: what the code looks like

You don't have to write this — ask Claude to write it for your specific PDF. Then run it locally.

import pdfplumber, csv # Open the PDF and extract the first table on each page with pdfplumber.open("inspections.pdf") as pdf: rows = [] for page in pdf.pages: table = page.extract_table() if table: rows.extend(table) # Write to CSV with open("output.csv", "w") as f: writer = csv.writer(f) writer.writerows(rows)

Ask Claude: "Write a pdfplumber script that extracts the table on page 3 of this PDF into a CSV. The table has these columns: [list them]."

Born-digital vs. scanned — it matters

✅ Born-digital PDF

  • → Text is embedded — copy/paste works
  • → PDFplumber extracts reliably
  • → Tables come out as structured data
  • → Fast even at scale

How to tell: try selecting text with your cursor. If it highlights cleanly, it's born-digital.

⚠️ Scanned PDF (image)

  • → No embedded text — it's a photo of a page
  • → Needs OCR before extraction
  • → Error rate depends on scan quality
  • → Validate everything against the original

Tools: Tesseract (free), Adobe Acrobat, ABBYY FineReader. natural-pdf handles this too.

Strategy 3: organize at scale

You have 500 PDFs. Before you can extract anything, you need to know what you have.

1
Build a manifest firstOne row per document. Filename, date received, document type, status. Never process the same file twice. Add columns as you learn more.
2
Classify before you extractUse AI to categorize documents into types (complaint, investigation, disposition). Extraction logic differs by type — don't mix them.
3
Exceptions are a second datasetEvery document your automation can't handle tells you something. Build a simple review tool rather than ignoring them.

What a manifest looks like

filename, doc_type, date_received, status, notes complaint_001.pdf, complaint, 2024-03-01, extracted, complaint_002.pdf, complaint, 2024-03-01, review, OCR unclear p.3 investigation_01.pdf,investigation, 2024-03-04, extracted, disposition_001.pdf, disposition, 2024-03-04, pending, unknown_file.pdf, UNKNOWN, 2024-03-04, manual, can't classify
  • → status column is the key: extracted / review / pending / manual / done
  • → Never delete rows — mark them with a status instead
  • → The manifest is your audit trail when an editor asks "how do you know?"

Strategy 3: classifying documents with AI

Classification prompt (run on each document) Read this document and classify it into exactly one of these categories: complaint, investigation, disposition, other. Return only the category name — nothing else. If you cannot determine the category, return "unknown".
Then validate I classified 500 documents into these categories. Here is a random sample of 20 with my classifications. Do you agree with each one? Flag any you'd classify differently and explain why.

Rule of thumb: validate 10–20% of AI classifications manually before you rely on them for a story. The error rate tells you whether to trust the rest.

Strategy 4: running AI locally

Why would you run a model on your own laptop instead of using ChatGPT?

  • 🔒 Sensitive documents — confidential sources, PII, ongoing investigations
  • 🔒 Organizational policy — some newsrooms prohibit sending work product to cloud AI
  • 🔒 Scale — processing 10,000 documents without hitting rate limits or paying per-token

How to do it: Ollama

  • → Free, runs on Mac/Windows/Linux
  • → One command to download a model: ollama pull llama3
  • → Works offline, nothing leaves your machine
  • → Models are smaller — good enough for classification, weaker for complex analysis

What local models are (and aren't) good for

✅ Good

  • → Classifying documents into categories
  • → Extracting structured fields from consistent templates
  • → Summarizing short documents
  • → Running the same prompt 10,000 times overnight

⚠️ Weaker

  • → Complex reasoning about document meaning
  • → Handling long documents (smaller context window)
  • → Tasks requiring nuance or legal/domain knowledge
  • → Anything you'd want a senior editor to fact-check

The sweet spot: use a local model for triage and classification at scale, then use a cloud model (or a human) for the cases that matter most.

Things that will go wrong

  • ⚠️ Hallucinated field values — AI fills in something not in the document. Fix: always ask for null on missing fields; spot-check against originals.
  • ⚠️ Substring collisions — "rick" matches "Kirkpatrick." Fix: word-boundary regex.
  • ⚠️ Wrong page crop — you're looking at the top when the data is at the bottom. Fix: test crop sizes on samples.
  • ⚠️ OCR misreads punctuation — (Warren) becomes CWarren). Fix: don't require literal punctuation in patterns.
  • ⚠️ Memory-only processing — one crash and you restart from zero. Fix: write to disk incrementally; build resume support.
  • ⚠️ No ground-truth check — errors go undetected. Fix: always verify totals against an official source.

Six things to remember

  • 1️⃣ Find the machine-readable layer first. Open your documents before writing any code. Page 3 of a TIFF might be clean text.
  • 2️⃣ Build a manifest early. One row per document, one column per thing you know. Never process the same file twice.
  • 3️⃣ Classify before you extract. Different document types need different extraction logic.
  • 4️⃣ Exceptions are a second dataset. Build a tool to review them — don't ignore them.
  • 5️⃣ Verify against something you already know. Official totals, published reports, anything.
  • 6️⃣ AI accelerates the coding, not the journalism. The judgment is still yours.
