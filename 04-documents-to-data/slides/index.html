The data is there. It's just trapped.

Court filings, inspection reports, financial disclosures — they all contain structured information locked inside unstructured formats.

📄 PDFs

Tables that look like tables but can't be copied. Fields that are always in the same place but have no column headers.

🖨️ Scanned docs

Images of pages. No embedded text at all. Need OCR before you can do anything with them.

📚 Hundreds of files

One document is manageable. Five hundred means you need a system — or you'll never find the needle.

Today: we'll use AI to read them, extract the data, and build a tool to search it — and AI will write most of the code.

vibecoding

You direct. AI writes the code.

You don't need to know Python or JavaScript. You need to know what you want — and how to tell AI when it got it wrong.

What you do

  • → Describe the problem clearly
  • → Run what AI gives you
  • → Tell it what broke or what's missing
  • → Repeat until it works
  • → Verify the output against the source

What AI does

  • → Writes the Python or JavaScript
  • → Debugs the errors
  • → Adapts to your specific document
  • → Explains what each part does
  • → Iterates as fast as you can describe

Step 1: Read & flag

You have a document. You don't know if it's worth your time. Let AI be the first reader.

Triage prompt Read this document as a journalist. List the three most significant facts a reporter should follow up on. For each one, note which page it's on and why it matters. Flag anything that looks inconsistent, missing, or unusual.
Comparison prompt Compare these two versions of this document. What changed? Focus on anything that shifts money, liability, deadlines, or who benefits. Ignore formatting differences.
1

Step 1: the rules

  • ✅ AI is the first reader, not the last. It flags things for you to verify — it does not verify them.
  • ✅ Ask for page numbers. If AI can't cite the page, it may be making up the fact.
  • ✅ Treat it like a tip from a source. You'd check that before publishing. Same rule.
  • ⚠️ Long documents exceed context limits. Break large docs into sections; ask AI to read them in chunks.
  • ⚠️ Know your org's policy before uploading sensitive documents to a cloud AI. When in doubt — extract locally.
2

Step 2: extract data locally

When you need a PDF's tables as real data — and you don't want to paste sensitive documents into a cloud tool.

◆ PDFplumber

Python library. Extracts text and tables from born-digital PDFs with high accuracy. Tables come out as lists you can push straight into a CSV.

◆ natural-pdf

Adds AI-assisted layout detection on top of PDFplumber. Better on complex column layouts and mixed content. Also handles scanned PDFs via OCR.

🔒 Both run entirely on your machine. Nothing is sent to a server. Matters for documents with sources, PII, or ongoing investigations.
⚠️ Born-digital vs. scanned: try selecting text in the PDF first. If it highlights cleanly, PDFplumber works great. If not, you need OCR first.
2

Vibecoding the extraction

Prompt to Claude or ChatGPT Write a Python script using pdfplumber that opens "inspections.pdf" and extracts the table on page 2. The table has these columns: facility name, address, inspection date, violations, pass/fail. Save the output as "inspections.csv". Explain what each section of the code does.
import pdfplumber, csv with pdfplumber.open("inspections.pdf") as pdf: table = pdf.pages[1].extract_table() # page 2 = index 1 with open("inspections.csv", "w", newline="") as f: csv.writer(f).writerows(table)

When it breaks: "I got this error: [paste error]. Don't rewrite the whole script. Tell me the minimal fix."

2

Define your schema before you extract

The tighter your schema, the more consistent the output — and the easier it is to catch when something goes wrong.

Inspection reports

  • → facility_name
  • → address
  • → inspection_date (YYYY-MM-DD)
  • → inspector_name
  • → violations (list)
  • → result (pass / fail / pending)

Court filings

  • → case_number
  • → filing_date (YYYY-MM-DD)
  • → plaintiff
  • → defendant
  • → charge_or_claim
  • → disposition
Rule: if a field isn't in the document, it should be blank — not guessed. Tell AI this explicitly every time.
2

Validate before you trust it

  • 1️⃣ Spot-check against originals. Pick 10–20 random rows. Open the PDF. Compare field by field.
  • 2️⃣ Check totals against something you already know. Official counts, published summaries, anything.
  • 3️⃣ Look for hallucinated values. AI fills in plausible-sounding things that aren't in the document. Null fields that should be null.
  • 4️⃣ The exception pile is often the story. Records the automation can't handle are worth looking at manually.

You don't need 100% accuracy to publish — you need to know your error rate and account for it.

Step 3: build the browse tool

You have a CSV. Now vibecode a local web app so you can actually search and report from it.

Prompt to build the tool Write a single self-contained HTML file that loads my CSV of inspection records and lets me: search across all fields as I type, sort by any column, click a row to see the full record, and see a count of matching results. No server, no build tools, no external dependencies — it should work by opening the file in a browser. Put the CSV path at the top so I can swap in my own file.
✅ You leave today with a working tool for your own documents. No server. No IT department. Just a file you open in Chrome.
3

Iterate until it does what you need

Each prompt adds one feature. You don't have to ask for everything at once.

Add filteringAdd a dropdown to filter by the "result" column (pass / fail / pending). Show the record count update as I filter.
Add exportAdd a button that downloads the currently visible rows as a CSV — so I can export just the records that match my search.
Add flaggingAdd a "Flag for follow-up" button on each record. Flagged records should appear highlighted and be exportable separately.

This is the same pattern used to build the review tools in the Cobb County ballot project — keyboard shortcuts, TSV export, resume support. All vibecoded.

What if you can't use a cloud AI?

Confidential sources, ongoing investigations, organizational policy — sometimes you can't paste documents into ChatGPT or Claude.

Run a model locally with Ollama

  • → Free. Runs on Mac, Windows, Linux.
  • → One command: ollama pull llama3
  • → Works offline. Nothing leaves your machine.
  • → Smaller models — good enough for classification, weaker on complex reasoning.
✅ Good for: classifying documents at scale, extracting structured fields from consistent templates, running 10,000 prompts overnight.
⚠️ Weaker at: long documents, nuanced analysis, anything you'd want a senior editor to fact-check. Use local for triage; use cloud (or a human) for the cases that matter most.

What to take home

  • 1️⃣ Find the machine-readable layer first. Open your documents before writing any code. The clean text may be hiding on page 3.
  • 2️⃣ Define your schema before you extract. Know exactly what fields you want. Blank beats hallucinated.
  • 3️⃣ Validate against something you already know. Official totals, published reports — catch errors before they're published mistakes.
  • 4️⃣ Build the tool, not the spreadsheet. A searchable HTML viewer is more useful than 500 rows you can't navigate.
  • 5️⃣ AI accelerates the coding, not the journalism. The judgment — what to look for, what it means, whether to trust it — is still yours.
